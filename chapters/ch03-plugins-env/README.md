## Goal

Build a small feature flag dashboard that demonstrates webpack plugins directly: generated HTML, copied static assets, and environment-specific values injected into the application bundle.

## What You Build

This chapter renders a feature flag dashboard whose title, release channel, API base URL, and enabled features change depending on whether webpack builds in development or production mode.

The chapter uses plugins for three distinct jobs:

- `HtmlWebpackPlugin` generates `dist/index.html`
- `CopyWebpackPlugin` copies `public/robots.txt` into `dist/`
- `DefinePlugin` replaces compile-time constants using values loaded from `.env.development` or `.env.production`

## Topics Covered

- webpack plugins
- generated HTML shells
- copied static assets
- environment variables with `dotenv`
- compile-time replacement with `DefinePlugin`
- Babel plugins with `@babel/plugin-transform-runtime`
- configuration validation with `webpack configtest`

## Project Structure

- `package.json`: scripts for development build, production build, config validation, and type checking
- `webpack.config.js`: mode-aware configuration, plugin wiring, and environment loading
- `babel.config.json`: Babel presets plus `@babel/plugin-transform-runtime`
- `tsconfig.json`: type-checking configuration
- `.env.development`: development-facing flag values
- `.env.production`: production-facing flag values
- `src/template.html`: HTML template consumed by `HtmlWebpackPlugin`
- `public/robots.txt`: copied into `dist/` by `CopyWebpackPlugin`
- `src/featureFlags.ts`: compile-time environment data and flags
- `src/loadBuildNotes.ts`: async note generation used by the app
- `src/renderFeatureDashboard.ts`: DOM rendering logic
- `src/types/env.d.ts`: TypeScript declarations for DefinePlugin constants

## How Environment Values Flow Into The App

1. `webpack.config.js` reads `.env.development` or `.env.production` based on `argv.mode`.
2. The config normalizes those values into strings and booleans.
3. `DefinePlugin` injects them as compile-time constants such as `__APP_ENV__` and `__FEATURE_INSIGHTS__`.
4. TypeScript source files consume those constants as if they were globals.
5. The final bundle contains the resolved values for the selected build mode.

This is not runtime environment lookup in the browser. The values are baked into the bundle during the webpack build.

## Why These Plugins Matter

`HtmlWebpackPlugin`

- generates the final HTML file automatically
- injects the built script into the page
- allows template variables such as the page title and environment label

`CopyWebpackPlugin`

- moves static files that do not belong in the module graph
- keeps files such as `robots.txt` out of your TypeScript source while still shipping them in `dist/`

`DefinePlugin`

- replaces identifiers at build time
- is ideal for feature flags, environment labels, and compile-time constants
- keeps environment handling explicit instead of hidden behind runtime heuristics

## Babel Plugin Choice

This chapter adds `@babel/plugin-transform-runtime` in `babel.config.json`.

Why it is here:

- it demonstrates how Babel plugins are configured separately from webpack plugins
- it helps Babel reuse helpers more cleanly instead of inlining them repeatedly across files
- it keeps the distinction clear: Babel plugins transform code, webpack plugins extend the build lifecycle

## Config Validation

The `config:test` script runs webpack configuration validation without doing a full normal build workflow. This is useful when you want to catch configuration mistakes early, especially once config files become more layered in later chapters.

## Expected Output

After `npm run build`, you should see at least:

- `dist/index.html`
- `dist/main.js`
- `dist/main.js.map`
- `dist/robots.txt`

After `npm run build:prod`, the HTML and JavaScript should reflect the production `.env` values instead.

## Verification Checklist

- `npm run build` succeeds
- `npm run build:prod` succeeds
- `npm run config:test` succeeds
- `npm run typecheck` succeeds
- `dist/index.html` is generated by webpack
- `dist/robots.txt` is copied from `public/`
- development and production builds contain different environment values

## Key Takeaways

- webpack plugins modify the build lifecycle, not just source-file transformation.
- Babel plugins and webpack plugins solve different problems and live in different config files.
- environment variables in browser bundles are usually compile-time replacements, not real runtime secrets.
- generated HTML and copied static assets should be part of the build story, not managed by hand.
